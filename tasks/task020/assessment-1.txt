产物不满意：
1. 提示词要求「页面标题栏或 favicon 上显示未读数量角标」，模型只通过 document.title 把未读数量写进了页面标题，完全没有在 favicon 上绘制角标的代码，favicon 角标功能缺失，只做了一半。
2. markAllOfTypeAsRead 函数只遍历了 state.active 中指定类型的通知，state.archived 里同类型的已归档通知完全没有被标记。而 markAllAsRead 却是同时标记 active 和 archived 的，同属"标记已读"类操作，行为不一致。用户在某个类型 tab 下点了"标记本组已读"后，归档区该类型的通知依旧显示未读。
3. addNotification 归档逻辑写成 `[{ ...oldest, archived: true }, ...newArchived]`，新归档的通知永远插入到 archived 列表最前面，导致历史通知 tab 中展示的顺序是最近归档的在最上方、最早归档的在最末尾，跟"历史通知"按时间从早到晚排列的正常直觉刚好反了。
4. notificationsUtils.test.js 中「should not mutate the original state」这个测试用例被放在了 describe('createNotification') 块内，但实际测试的是 addNotification 函数的不可变性，测试组织和函数名不匹配。

过程不满意：
1. 模型一上来就直接创建了 7 项 TodoList，然后才开始读文件了解项目结构。没有先分析需求再制定计划，违反了标注规范中"不能刚收到任务就直接创建 TodoList"的要求。
2. 探索阶段读了 Dashboard.jsx 和 kanban 目录下的多个文件，这些跟通知中心任务没有任何关系，属于不必要的文件读取。
3. 测试写完后首次运行就失败，模型没有分析是函数实现有毛病还是测试的预期错了，直接选择修改测试的断言值去迁就实现。从思考过程看，归档顺序的期望值改了两次才全部通过，典型的测试向代码妥协。
4. App.jsx 的路由注册过程反复折腾了至少 4 轮——第一次 edit 产生了重复行，接着 Write 重写整个文件，之后又因为"外部修改"重新 edit 导入和路由，从思考记录第 214 行一直纠缠到第 796 行才收尾，同一个简单操作拖了四五百行，效率极低。
5. lint 问题分了三次才修干净：第一次跑 lint 修了一部分再跑，第二次又冒出新的再修，第三次调整 prefsRef 的访问方式才消停。没有在第一次 lint 后就全面排查所有问题。
6. 模型在最终总结中写「运行 npm run dev 即可从首页点击进入页面体验」，但全程没有执行过一条 npm run dev 命令，也没有打开过浏览器做任何页面验证，根本没实际确认页面能不能正常工作，属于虚假完成。
