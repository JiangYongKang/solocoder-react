1. 模型在修改 App.jsx 注册路由的步骤中，尝试了两次 edit_file_search_replace 均失败后，第三次 view_files 读到 App.jsx 时错误地声称"App.jsx 中已经注册好了 MindMapPage 的导入、任务卡片和路由"，实际上 App.jsx 中没有任何 MindMapPage 的 import、TASKS 卡片条目和 Route 配置。导致思维导图页面无法从首页通过点击卡片跳转访问，提示词第 8 条"在 src/App.jsx 中注册该页面的路由，确保从首页点击卡片能正确跳转"完全没有完成。根本原因是模型在文件编辑失败后产生了幻觉，未实际核查 App.jsx 中是否存在目标代码。正确做法是逐行检查 App.jsx 的 imports、TASKS 数组和 Routes 部分，确认缺失后手动添加对应的 import、卡片和路由条目。

2. 模型在 index.jsx 中实现 handleContextMenu 函数时，只判断了 selectedId 是否存在，没有判断右键事件的目标元素是否为画布上的节点，导致在已选中任意节点的状态下，右键点击页面工具栏、标题栏或空白区域时也会弹出节点操作菜单。业务影响是用户在做非节点操作（如点击导出按钮）时误触右键会看到与当前交互无关的弹窗。根本原因是将上下文菜单的触发逻辑绑定在整个页面 div 上而不是限定在画布节点区域。正确做法是让 contextmenu 事件只在画布节点元素上触发，或在页面级 handler 中增加对事件目标是否为 mind-node-group 元素的判断。

3. 模型在 index.jsx 的 handleAddChild 和 handleAddSibling 函数中，将 setSelectedId(newNodeId) 和 setEditingId(newNodeId) 放在了 setTree 的 updater 回调函数内部。在 React 中，在 setState 的 updater 回调里调用另一个 setState 是反模式，虽然 React 的自动批处理会在大多数情况下掩盖问题，但这种写法不符合 React 最佳实践，代码可读性和可维护性差。正确做法是将这些状态更新放在 setTree 的回调外部，让 React 正常批处理即可。

4. 模型在 mindMapCore.js 的 findParentNode 函数中，for 循环体内的条件判断 if (found !== null || (found === null && child.id === nodeId)) 的第二部分 (found === null && child.id === nodeId) 是死代码，永远不会为真。因为当 child.id === nodeId 时，递归调用 findParentNode(child, nodeId, root) 一定会返回 root（非 null），即 found 不为 null。虽然不影响功能正确性，但说明模型写这段逻辑时没有充分推演递归调用链，代码产物不干净。正确做法是简化为 if (found !== null) return found。

5. 模型在 MindMapCanvas.jsx 的 handleWheel 函数中，缩放边界值写死为 Math.max(0.3, Math.min(2.0, ...))，而不是使用从 constants.js 导入的 MIN_ZOOM（0.3）和 MAX_ZOOM（2.0）常量。如果将来需求变更需要调整缩放范围，只修改 constants.js 中的常量值并不会反映到 handleWheel 的实际行为中，存在维护风险。正确做法是在 handleWheel 中引用 MIN_ZOOM 和 MAX_ZOOM 常量。

6. 模型实现的节点间连线使用了三次贝塞尔曲线（SVG C 路径命令），而提示词第 1 条明确要求"节点之间用直线或折线连接"。虽然贝塞尔曲线在视觉效果上更好，但产物偏离了需求规格。正确做法是按照需求文档使用 Line（直线）或 Polyline（折线）实现连线。
