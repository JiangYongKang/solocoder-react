产物不满意：

shiftFormulaReferences 函数生成的包含 #REF! 的公式字符串（如 =#REF!+A1+A2）无法被现有的公式求值器正确解析。在 tokenize 函数中，字符 # 不匹配任何 token 分支而被跳过，REF 被当作 ident 类型 token，parseFactor 对 ident 类型无匹配处理直接返回 0，导致被删除引用的公式计算出错误的数值结果（0+20+30=50）而非显示 #REF! 或 #ERROR! 错误标记。根本原因是 shiftFormulaReferences 输出的 #REF! 标记格式与现有的公式分词器/求值器之间缺少接口契约——新增功能输出的格式旧模块不认识。正确做法是让 tokenize 函数增加对 #REF! 和 #ERROR! 等特殊错误标记的识别分支，或在 evaluateTokens 中检查 ident token 名称是否属于错误标记并返回对应的错误值。

deleteRow 和 deleteCol 的公式引用更新测试只验证了 raw 字段未验证 display 计算结果。测试用例"deleteRow - 应该同步更新公式中的行引用"只断言 result.cells.B2.raw 为 =#REF!+A1+B1，缺少对 display 的断言。同样的，"deleteCol - 应该同步更新公式中的列引用"也只检查 raw 未检查 display。这导致上述 #REF! 求值错误的缺陷未被测试捕获——如果补充 display 断言将因实际输出为数值 50 而非错误标记而失败。根本原因是编写测试时只验证了表面字符串产出，未验证最终业务结果（公式计算结果的正确性）。正确做法是为所有行列操作测试同时断言 raw 和 display 字段，覆盖正常更新和被删除引用两种场景。

saveToLocalStorage 被 useEffect 以整个 state 对象为依赖（index.jsx 第 523-525 行），每次 setState 调用都会触发完整的 localStorage 写入。insertRow/deleteRow/insertCol/deleteCol 等操作每次都返回新的 state 对象引用，这意味着每次单元格编辑、行列操作都会触发 JSON.stringify 序列化整个 cells 字典。默认 50 行 x 26 列规模下每次操作都全量写入并非严重性能问题，但当电子表格数据量增大时这会成为瓶颈。根本原因是持久化没有加防抖（debounce）限制频率。正确做法是用 useRef 配合 setTimeout 实现 debounce，在连续操作结束后才执行一次 saveToLocalStorage。

工具栏中的"加粗 (Ctrl+B)"和"斜体 (Ctrl+I)"按钮 tooltip 原来已有快捷键提示，模型在 handleKeyDown 中补充了 Ctrl+B 和 Ctrl+I 的实现，修复了快捷键不可用的问题。但 handleKeyDown 的依赖数组（第 346 行）中包含了 toggleBold 和 toggleItalic，这两个函数本身依赖 activeCell、cells、applyStyle，而 applyStyle 依赖 selection。由于 selection 频繁变化，handleKeyDown 的 useCallback 会频繁重新创建，但这属于第一轮已有的设计问题，本轮未恶化。快捷键功能本身实现正确。
