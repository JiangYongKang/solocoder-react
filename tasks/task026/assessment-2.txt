产物不满意：
1. TaskList.jsx 的 EditableCell 组件中 handleCommit 校验逻辑存在字段错误键不匹配的 BUG。validateTask 函数在校验日期顺序时，当 startDate 晚于 endDate，错误仅记录在 errors.endDate 键上。而 handleCommit 中只检查 result.errors[field]（即当前编辑字段），当用户在 startDate 的 EditableCell 中输入一个晚于 endDate 的日期时，validateTask 返回 errors.endDate 而非 errors.startDate，导致 if (!result.valid && result.errors[field]) 条件为 falsy，校验被静默跳过，用户可以在不看到错误提示的情况下提交不合法的日期顺序。根本原因是 validateTask 的 date order 校验只将错误绑定到 endDate 字段，而 EditableCell 只检查当前编辑字段的错误键，两者未对齐。正确做法是 validateTask 的 date order 错误应同时记录在 startDate 和 endDate 两个键上，或者 EditableCell 在校验失败且当前字段无错误时退而检查整个 errors 对象是否有任何错误。
2. Timeline.jsx 的拖拽实现中，mousemove 事件处理函数每帧都通过 setDragging 更新 currentX 到 React state 中，而 useEffect 的依赖数组包含 dragging 对象。这意味着每次 mousemove 触发 setDragging → 组件重渲染 → useEffect 清理函数移除旧事件监听器 → useEffect 重新添加新事件监听器。在拖拽过程中每帧都会执行一次完整的 listener 拆卸与重装，产生不必要的 GC 压力和潜在的事件处理间隙。根本原因是将高频变化的 currentX 放在 React state 中驱动了不必要的 effect 重建。正确做法是用 useRef 存储 drag 过程中的实时坐标，只在 drag 开始和结束时通过 setState 触发渲染。
3. calculateBarPosition 函数内部仍然手动使用 diffDays × dayWidth 计算 left 和 width，未重构为调用 dateToPx 函数。虽然提示词第 5 条要求"清理或实际使用"这两个函数，模型选择了"实际使用"并在 Timeline.jsx 中确实调用了它们，但 calculateBarPosition 作为位置计算的核心函数，其内部逻辑与 dateToPx 高度重复：dateToPx 内部就是 diffDays × dayWidth，calculateBarPosition 将同样的计算内联了一遍。模型在引入 dateToPx 工具函数后没有追溯重构调用方，留下了逻辑冗余。根本原因是修改时只关注了"在组件中使用"而非"在项目中消除重复逻辑"。正确做法是 calculateBarPosition 内部直接用 dateToPx 替代手动 diffDays × dayWidth 计算。
4. getDefaultTasks 构建的默认层级只有两层：根任务（parentId=null）和直接子任务。没有任何子任务有进一步的孙任务。而模型在总结中声称"9 个任务现在具有清晰的 3 级层级结构"，实际代码中并没有 depth=2 的可见任务。虽然提示词只要求"设置合理的父子关系"而未指定层级深度，但模型总结中的"3 级层级结构"描述与代码实际不符，存在虚假声称。根本原因是模型在总结开发成果时没有审阅实际产物内容，凭印象做出断言。正确做法是总结时应基于 getVisibleTasks 实际输出的 depth 分布来验证层级结构的深度。

过程不满意：
1. 模型在收到提示词后，第一时间就调用 todo_write 创建了 8 项待办任务，列出每项对应的问题编号，然后才开始读取文件分析代码。这违反了开发流程的基本顺序——应先通过读取全部相关文件来理解当前代码状态，形成修复方案，再规划执行步骤。先规划后分析导致最初的 TodoList 只是一个机械地将提示词逐条翻译为待办项的列表，没有反映基于代码真实情况的技术方案。根本原因是对任务规划的理解停留在"翻译需求为任务项"的层次。正确做法是先读取所有涉及的源文件，分析各问题的根因和修复范围，然后制定包含技术方案的 TodoList。
2. 模型在修改 Timeline.jsx 的拖拽逻辑以接入 pxToDate 时，出现了两次修正。模型在思考轨迹中自述"我发现 move 模式的拖拽逻辑有个小 bug，需要修复。让我简化并确保 pxToDate 正确用于所有拖拽模式"，然后接连使用了两次 edit_file_search_replace 来修正同一段拖拽逻辑。这是典型的试错式开发——第一版实现不正确，通过后续操作修补。根本原因是修改前没有完整推演拖拽三种模式（move/resize-left/resize-right）下 pxToDate 的坐标转换是否正确。正确做法是在动手修改前先画出坐标转换的公式，确保所有分支的公式推导无误后再一气呵成地实施。
3. 模型在总结开发成果时存在多处与实际不符的数字描述。声称"新增 33+ 测试用例"，实际新增了 36 个（7+8+9+9+3）；声称"92 个全部通过（原 59 + 新增 33）"，实际原测试数 56 + 新增 36 = 92，原测试数量也被算错；声称"3 级层级结构"，实际只有 2 级。这些错误都表明模型在写总结时没有对照实际文件和运行结果来核实数字，凭记忆或估算得出，存在虚假完成的特征。根本原因是总结时没有回头验证实际产物。正确做法是在完成所有修改后，重新读取关键文件的测试用例数、代码的层级结构等客观数据，确保总结中引用的数字和描述与实际产物一致。
