产物不满意：
1. sanitizeUrl 函数中存在对已转义 URL 的二次转义问题。sanitizeUrl 在判断 URL 安全后会调用 mdEscapeHtml(url) 进行转义，但函数调用的上下文是 mdProtectInline，而 mdProtectInline 的入参已经是经过 mdEscapeHtml 转义后的内容。以链接 Markdown [text](https://example.com?q=<x>) 为例：整行先被 mdEscapeHtml 转义后变成 [text](https://example.com?q=&lt;x&gt;)，然后链接正则捕获到的 linkUrl 已经是 https://example.com?q=&lt;x&gt;，sanitizeUrl 拿到这个 URL 后再次调用 mdEscapeHtml，把 &lt; 中的 & 转成 &amp;lt;，最终 URL 变成 https://example.com?q=&amp;lt;x&amp;gt;，导致链接指向的目标 URL 与用户预期不一致。根本原因是在设计数据流时没有理清 sanitizeUrl 的输入数据已经在调用链上游做过一次 HTML 转义，未区分"原始 Markdown 中的 URL"和"已转义内容中的 URL"。正确做法是 sanitizeUrl 只对原始未转义的 URL 进行转义，或者在调用 sanitizeUrl 之前先从 rawLine 而非 escapedContent 中提取 URL。

过程不满意：
1. 模型在重写 renderMarkdown 的过程中经历了多轮试错式调试。模型在第一次写完代码后运行测试发现了 5 个问题：引用符号 > 被转义成 &gt; 导致正则匹配不上、行内元素内容双重转义、测试断言需调整等。模型没有先分析这些问题是否都源于同一个根本原因即转义顺序与正则匹配的冲突，而是逐一零散修改：先改引用正则→测试失败→再改测试断言→测试失败→再改链接文本转义→发现双重转义→又撤销修改。反复了约 3-4 轮才通过所有测试。同一类转义顺序问题反复出现两次以上，说明模型在第一批修改前没有系统性分析"先全量转义再匹配正则"这种架构在哪些场景下会出问题。根本原因是对复杂函数的架构设计缺乏前置推演，采用了"写出代码再靠测试结果驱动修正"的试错方式。正确做法是在动手写代码前，先画出数据流转图，明确哪些步骤用原始文本匹配正则、哪些步骤用转义后文本做行内解析，确认不存在循环依赖后再编码。
2. 模型在最终总结中声称"第三轮所有 5 个问题已全部修复完成"，但提示词第 5 条（还原 App.jsx 无关修改）全程没有被处理，模型既没有读取 App.jsx 也没有编辑 App.jsx。这一声称与实际情况不符，属于虚假完成。根本原因是模型在写总结时没有对照提示词逐条核实产出，仅凭记忆或感觉判断完成度。正确做法是在写总结前重新回看提示词的每一条要求，逐一确认是否已处理并有对应的产物证据。
3. 模型启动 dev server 后仅描述了"已启动"和"可以打开浏览器验证"，但提示词第 6 条明确要求"确保所有子页面在 375px 和 1440px 宽度下布局不崩坏，需要启动 dev server 实际验证"。模型没有展示在不同宽度下的截图、也没有用浏览器 DevTools 的设备模拟功能实际切换到 375px 和 1440px 进行验证的过程记录。COT 中只看到启动 dev server 和打开预览两个操作，验证环节缺失。根本原因是把"启动 dev server"等同于"完成了布局验证"，未理解提示词要求的"实际验证"是需要有可观测的验证动作和结果。正确做法是分别切换到 375px 和 1440px 宽度截图记录，或至少用文字描述页面在各宽度下的布局表现，证明关键元素没有溢出、断裂或隐藏。
