1. 在刚收到任务创建 TodoList 时，模型只规划了4项任务（重构 findActiveExamDraft、动态 MAX_ATTEMPTS、修改 index.jsx、补充测试），但提示词共有5条需求，第4条"还原 App.jsx 中的 MindMapPage 等无关变更"未被列入 TodoList 的任何一项。这直接导致整个开发过程中模型从未查看 App.jsx 文件，该需求被彻底遗漏，最终产物中 App.jsx 的 MindMapPage 相关代码毫发未动。根本原因是规划阶段缺少逐条遍历提示词的环节，在未完整理解所有需求前就仓促创建了 TodoList。正确做法是先通读全部提示词条目，逐条转化为 TodoList 中的可执行任务项后再开始编码。
2. 在第一次修改 index.jsx 将草稿恢复从 useMemo 迁移到 useEffect 时，模型的方案是将 getInitialFromUrl 放在组件渲染流程中直接调用并赋值给变量，配合多个独立 useState 和 useEffect 组合。运行 ESLint 后触发 setState-in-effect 错误，模型不得不将方案回退为 useState 惰性初始化。这一来一回消耗了一轮额外的修改-测试-返工时间。根本原因是修改 React 组件状态初始化逻辑前未充分评估 React Hooks 规则与 ESLint react-hooks 规则的约束关系，未能在一开始就识别出正确的模式分工。正确做法是动手重构前先明确：纯读操作应放在 useState 惰性初始化函数中执行，副作用操作（如 cleanupExpiredExamDrafts）单独放在 useEffect 中，两者职责分离而不交叉。
3. 在执行 pnpm run test 全量测试后终端输出显示有7个非考试模块的测试失败（属于 snippets 和 spreadsheet 模块），模型直接声称这些是"预先存在的历史问题，与本次修改无关"，但未通过 git stash 暂存当前修改后对比全量测试结果、也未执行 git diff 确认改动范围是否触及这些模块。虽然当前代码库的全量测试已全部通过（32 文件 2103 测试），但模型在当时的上下文中不做版本控制验证就直接下结论是不严谨的。根本原因是缺乏通过版本控制工具做因果关系隔离验证的意识，用模块归属推断代替了实际的 diff 对照。正确做法是先 git stash 保存当前修改，运行测试记录基线结果，再 git stash pop 恢复修改后运行测试对比，确认失败用例在修改前后保持一致后才可宣称是预先存在的问题。
4. 提示词第5条要求"确保所有子页面在 375px 和 1440px 宽度下布局不崩坏，需要启动 dev server 实际验证"，模型启动了 dev server 并打开了预览，但仅检查了 VS Code diagnostics 有无编译错误，未使用浏览器工具在 375px 和 1440px 两种视口宽度下对考试系统的题库管理、组卷考试、考试作答、成绩查看、成绩记录等所有子页面逐一进行截图或交互验证，也未记录任何具体的验证行为和观察结果。根本原因是将"dev server 无编译错误"等价于"布局验证通过"，未区分编译期正确性与运行时响应式表现。正确做法是使用浏览器工具分别设置为 375px 和 1440px 宽度，对考试系统各个子页面的关键交互状态逐一截图验证，确保无元素溢出、重叠或截断。
