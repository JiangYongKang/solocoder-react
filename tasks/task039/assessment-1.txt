产物不满意：
1. index.jsx 中正则输入框使用普通 input 元素实现，没有语法高亮功能，违反了提示词第 1 条"上方是正则表达式输入框（带语法高亮）"的要求。根本原因是模型在需求分析时遗漏了"语法高亮"这一技术要求，直接将正则输入作为普通文本输入框实现。正确做法是使用 contenteditable 或 code mirror 等方式为正则表达式中的特殊字符（如转义符、字符类、量词等）应用不同颜色着色。
2. index.jsx 中历史记录需要用户手动点击保存按钮才会记录，而不是"每次测试后自动保存"，违反了提示词第 6 条"每次测试后自动保存正则表达式和测试文本到历史列表"的要求。根本原因是模型将"测试"理解为点击保存按钮的操作，而不是用户输入正则后实时匹配即为"测试"。正确做法是在正则或测试文本变化且匹配结果更新后自动将当前正则和文本保存到历史中。
3. index.jsx 中替换 diff 算法实现粗糙，仅从第一个差异字符切割原文本和替换文本，不能精确标记文本中间部分的变更区间，差异高亮标记不准确。根本原因是模型使用了简化的字符串比较逻辑，没有实现真正的逐行或逐词 diff 算法。正确做法是实现能够标记多个变更区间的 diff 算法，对每个变更区间独立进行高亮。

过程不满意：
1. 模型在最终总结中声称"正则输入框带语法高亮"，但实际产物中正则输入框是普通 input 没有任何语法着色，属于虚假完成声明。根本原因是写总结时没有重新审阅实际产物内容。正确做法是总结时必须基于实际文件内容而非记忆。
2. 模型在开发初期没有先完整分析 7 条提示词需求再规划技术方案，而是直接开始编码，导致语法高亮和自动保存两项需求被遗漏。根本原因是没有在开发前建立需求检查清单。正确做法是先将所有需求逐条转化为实现任务清单，确保每个需求都有对应的实现计划。
3. 模型在 build 失败后没有启动 pnpm run dev 来验证页面在实际浏览器中是否可访问和功能是否正常，缺失运行时验证环节。根本原因是模型认为构建和测试通过即等同于功能可用。正确做法是开发完成后启动 dev server 手动验证关键功能路径。
