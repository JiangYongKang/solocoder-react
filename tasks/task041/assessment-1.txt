产物不满意：
1. getDisabledValues 函数在初始状态（所有 SKU stock 为 0、selection 为空对象）时，会把所有规格组的全部规格值都标记为禁用，导致预览区所有按钮都处于灰色不可点击状态，用户无法开始选择。根因是函数在计算禁用集合时，没有先检查当前 selection 中是否已有除当前组以外的其他已选规格值——提示词要求的是"当用户选中某个规格值后"才联动禁用，初始无选择时不应禁用任何按钮。正确做法是在 getDisabledValues 开头增加判断：如果 selection 为空或者只有当前组这一个待计算的组有值（实质上等同于无前置选择），直接返回空对象，不要进入禁用计算逻辑。
2. PreviewPanel 的图片展示区域将所有匹配图片通过 position:absolute 全部叠加在同一个 240x240 容器中，由于完全重叠导致用户无法看清任何一张图片的内容，不符合"主图区域自动切换显示匹配的图片组合"的展示意图。同时，当前 getImagesForSelection 只返回有图片的选中值，未实现提示词要求的"如果某个规格值未上传图片，则保持当前图片不变"的上下文保留逻辑——例如先在"颜色"中选择红色（已配图）、再选择尺寸 S（无配图）时，红色图片应继续显示而不是消失。根因是图片展示采用了简单叠加方案而非切换 + 保留机制。正确做法是为每个规格维度的图片单独分配展示区域，或用状态追踪上一次有效图片，在新选择无图片时回退到已有图片。
3. SkuListPanel 中 handleSkuStockChange 和 handleSkuPriceChange 在处理用户清空输入框的场景时，将 stock 和 price 赋值为空字符串 '' 而非数字类型。例如用户先输入 100 再全部删除，SKU 对象的 stock 字段会变成 ''，与 generateSkuList 默认生成的数字 0 类型不一致。虽然在 PreviewPanel 中通过 != null 判断容错绕过了渲染崩溃，但数据类型约定被破坏。根因是输入值空字符串的转换逻辑把 '' 透传进了数据模型。正确做法是空字符串统一转换为 0。
4. SkuSelectorPage 中 groups 变化时无条件清空 selection 的 useEffect 过于激进：用户仅修改了规格组名称或规格值名称（不改变 ID 映射关系）时，selection 引用变化触发清空，导致用户已做的规格选择丢失。例如用户把"红色"改名为"Red"时，之前在预览区选择的所有规格都会被重置。根因是 effect 没有判断 groups 的实质变化是否真正影响了 specs 映射关系。正确做法是在 effect 中对比新旧 groups 的 id/valueId 集合，只有集合变化时才清空 selection。
5. DEFAULT_SPEC_GROUPS 中的规格组和规格值使用了硬编码的字符串 ID（如 "default_group_color"、"default_color_red"），与用户通过 generateId 动态添加的组/值 ID 格式（如 "group_1700000000000_1"）不一致。虽然功能上不产生错误，但数据模型不统一，后续如果基于 ID 做判断或迁移会有隐患。根因是默认数据未通过相同的工厂函数创建。正确做法是 DEFAULT_SPEC_GROUPS 也通过 createSpecGroup / createSpecValue 或统一的 generateId 生成 ID。
6. 测试文件 skuCore.test.js 中 syncSkuList 相关用例的第 268 行和第 279 行传递了三个参数：syncSkuList(oldSkus, newSkus, mockGroups)，但 syncSkuList 函数签名只接受两个参数 (oldSkus, newSkus)。JavaScript 虽然会忽略多余参数使测试恰好通过，但这说明测试编写时未对照函数签名校验，测试的虚假通过掩盖了调用不一致的问题。根因是写测试时以为函数需要三个参数但实际只需要两个。正确做法是移除第三个多余参数，使测试调用与函数签名一致。

过程不满意：
1. 模型在收到任务提示词后第一步就创建了 TodoList 包含 10 个子任务，但在此之前只执行了查看项目根目录和 src 目录两个操作，没有仔细阅读提示词的每一条需求、没有分析项目现有页面的代码结构、没有评估任务复杂度就急于规划。根因是对任务缺乏充分的前置理解就直接进入规划阶段。正确做法是先完整阅读提示词每一条需求、参考现有页面的实现模式（如 App.jsx 路由注册方式和已有页面的目录结构），在充分理解后再制定执行计划。
