产物不满意：
1. ScheduleGrid.jsx 中星期表头未实现垂直方向 sticky 固定。schedule-planner.css 第186行定义了 `.schedule-grid-header` 类并设置了 `position: sticky; top: 0`，但该 CSS 类在 ScheduleGrid.jsx 所有 JSX 元素中从未被使用，实际表头渲染用的是 `.schedule-day-header` 类，而该类仅设置 `position: relative`，不含任何 sticky 定位。此外，`.schedule-grid-header` 本身使用了 `display: contents`，即使该 class 被应用也无法实现 sticky 固定，因为 `display: contents` 的元素没有自身盒模型。这导致垂直滚动时星期表头随课表一同滚出视野，不满足需求第1条"表头（星期行）在垂直滚动时固定可见"。正确做法是为每个 day-header 单独设置 `position: sticky; top: 0; z-index: 10`，或者用一个包裹容器实现头部固定。

2. PrintView.jsx 中课程跨时段显示使用了 HTML table 的 `rowSpan` 属性，但逻辑实现不完整。代码在循环渲染 TIME_SLOTS 时，对每个时间段行都无条件生成了完整的7列 `<td>` 单元格，同时在课程所在的 `<td>` 上设置了 `rowSpan: spanDuration`。在 HTML table 中，rowSpan 要求被跨越的后续行不再渲染同一列的单元格，否则表格会出现多出的单元格导致列错位和布局变形。当前实现下，课程占据第2节-第3节共2个时段时，第3节行仍会生成该列的 `<td>`，造成打印视图中表格单元格重复、列宽不对齐。正确做法是跟踪哪些单元格已被上方 rowSpan 跨越，在对应的时段行中跳过已被跨越的列。

3. ScheduleGrid.jsx 中当前时间行高亮逻辑不完全符合需求。第101行判断条件为 `slotIdx === currentSlot && dayIdx === currentDay`，这导致只有当前时间段与当前星期交叉的那个单一单元格被高亮（`.schedule-grid-cell.current`），而其余6列的同时间段单元格没有任何高亮。需求第1条明确要求"当前时间对应的行高亮浅色背景"，意味着该时间段整行（所有7列）都应有浅色背景。时间标签列虽然正确高亮了（第81行仅判断 slotIdx），但网格区域7列中只高亮了1列，与需求描述不一致。正确做法是将判断条件改为仅 `slotIdx === currentSlot`，使整行网格单元格统一高亮。

4. index.jsx 第231-233行定义了 `handleScheduledCourseDragStart` 函数，用于已排课程在课表中的拖拽操作。但这个函数在文件中仅被定义，既没有通过 props 传递给 ScheduleGrid 子组件，也未在 index.jsx 的其他地方被调用，ScheduleGrid 中的已排课程卡片也没有设置 draggable 属性和拖拽事件处理器。这是一个未连接的废弃函数，属于代码遗留产物。应该要么实现已排课程在课表中的拖拽调整功能，要么清理该废弃函数。

5. PrintView.jsx 第102-131行在打印视图的课表网格下方额外渲染了"课程列表"卡片区域，展示了所有课程的名称、教师、教室、时长、学分。需求第6条要求打印视图"隐藏编辑UI仅保留课表网格"，课程列表属于编辑/管理UI的一部分，在打印输出中出现会干扰课表的纯净性，而且这部分内容与课表网格数据重复，浪费纸张空间。正确做法是打印视图中只保留课表网格 table，移除下方课程列表区域。

过程不满意：
1. 模型在写完测试文件 scheduleUtils.test.js 后首次运行 `pnpm test schedule-planner`，终端立即报出14处语法错误：包括6处括号不匹配的错误（第150行、238行、380行、385行、386行、402行缺少闭括号），数组闭合使用了错误的 `}` 而非 `]`（第551行），forEach 回调参数语法错误（第664行），测试断言调用结构错误（第568行）等。这14处全是基础的语法低级错误，模型随后连续执行了8轮 edit_file_search_replace 来逐个修复这些语法问题。模型在批量写完全部76个测试用例后，没有做任何静态检查就直接运行测试，导致需要花费大量轮次修补低级错误。正确做法是每写完一批测试用例（如一个 describe 块），先目视检查括号配对、字符串闭合和基本语法，确认无误后再继续下一批。

2. 模型在编写 detectAllConflicts 冲突检测测试时，连续重写了3次测试逻辑才使测试通过。第一次错误地期望 `typeof c.duration` 匹配字符串（实际 duration 是数字），被测试驳回后改成直接比较数字。第二次构造测试数据时期望冲突数量为2，但实际只产生了1个冲突，因为测试数据中两门课程虽有教室冲突和教师冲突的可能，但没有正确设置教师为同一人——模型又做了一轮修改才正确构造出同时包含教室冲突（教室A相同）和教师冲突（同一教师）的测试场景。这说明模型在编写测试时没有先阅读 `detectAllConflicts` 函数内部的冲突判定条件——需要同一天、周次兼容、且时段重叠才会产生冲突——而是凭感觉设计测试数据，靠反复运行测试来试凑期望值。正确做法是先阅读被测函数源代码，理解其冲突判定的三个前置条件，再精确构造满足所有条件的测试数据一次性写对测试。
