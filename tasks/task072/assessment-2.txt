产物不满意：
1. inferObjectFieldSchemas 函数中添加的递归深度保护存在过度抑制数组类型信息的问题。当 depth + 1 > MAX_RECURSION_DEPTH（即递归深度超过 50 层）时，代码完全跳过了 info.hasArray 分支内的数组项类型推断逻辑，导致该层级的数组字段其 baseType 保持为空字符串，最终 fallback 为 'any'，而非至少标注为 'any[]' 或 '(unknown)[]'。例如深度为 51 层的 `{"items": [1,2,3]}` 会生成 `items: any` 而非 `items: number[]`，完全丢失了该字段是数组这一基础类型信息。根本原因是模型将"阻止递归展开嵌套类型"与"标注是否为数组类型"两个独立语义混在一起做了统一跳过处理。正确做法是在超过递归深度限制时仍应保留数组包装标记，将类型设置为 'any[]' 或类似形式。
2. collectNestedObjects 函数在深度限制处提前返回时直接返回 typeDefs 数组而非与其他分支一致返回 `{ typeDefs, ... }` 结构的对象。当深度达到限制时（depth > 200 或 depth > 50），函数返回的是裸数组，调用方通过 result.listTypeName 或 result.topLevelListName 访问时得到 undefined，导致对应字段的类型名称未正确设置。由于提前返回发生在深度边界上，大多数正常深度的数据不会触发，但在边缘场景下会产生不一致的返回类型。原因是函数内部多个 return 路径的返回形状不一致，正确做法是在所有 return 路径统一返回相同结构的对象。
