产物不满意：
1. annotatorCore.js 的 createAnnotation 用 Date.now()+Math.random() 生成标注 ID，Math.random() 仅约 2^53 种可能值，同一毫秒内连续快速创建时存在 ID 碰撞风险。测试在独立时序下不会触发，但实际交互中双击连续创建会导致 ID 重复，后续操作会误作用到错误元素。应使用递增计数器或 crypto.randomUUID() 确保全局唯一。
2. annotatorCore.js 的 pushHistory 在撤销后再做新操作时通过 slice(0, historyIndex+1) 截断 future 历史，但逻辑有撤销后和普通添加两个分支，截断与添加不统一，增加维护复杂度。应在 historyIndex < history.length-1 时始终截断 future 历史再 push 新状态，统一一条路径处理。
3. storage.js 的 saveAnnotation 用 localStorage.setItem 直接写入。当 annotationData 的 Base64 超 5MB 限制时 JSON.stringify 成功但 setItem 静默失败，用户以为保存成功实际已丢失。代码只做了简单大小判断，未处理整体序列化后仍可能超限的情况。应在 setItem 后用 try-catch 捕获 QuotaExceededError 并提示用户。
4. index.jsx 的 Canvas 鼠标事件处理使用了 e.clientX - canvas.getBoundingClientRect().left 计算鼠标在 Canvas 上的位置。当 Canvas 应用了 CSS transform（如缩放）或页面发生滚动时，getBoundingClientRect 的值可能不准确导致鼠标和标注元素的交互偏移——用户点击的位置与实际被操作的元素产生位置偏差。应该使用 canvas.width 和 canvas.height（Canvas 内部分辨率）配合 offsetX/offsetY 或计算 scale 因子来做坐标转换。
5. index.jsx 的 handleKeyDown 处理了 Delete 和 Backspace 键删除选中元素，但没有根据标注的 selectedTool 做区分处理——当用户当前选中的是画笔工具而非选择工具时，按 Delete 键应该是删除画笔绘制的最后一个路径段而非删除选中的标注元素。当前实现无视工具状态统一执行标注元素删除，导致画笔绘制过程中误触 Delete 会意外删除之前绘制的完整元素。应该在删除操作中判断当前工具类型，只在选择工具或元素已选中时执行删除。

过程不满意：
1. 模型在开始实现前读取了 image-editor 和 whiteboard 两个已有模块的代码作为参考。image-editor 确实与截图标注功能有相似之处（Canvas 操作、图层管理），但 whiteboard 是一个在线协作白板，其交互模式（多用户同步、网络层）与截图标注完全无关，读取 whiteboard/index.jsx 是无效参考。模型的探索范围偏大，应该只选择功能最接近的参考模块。
2. 模型首次运行 annotatorCore 测试遇到 5 个失败，涉及 pointNearLine、pointNearRectEdge、resizeAnnotation 和 pushHistory。逐函数分析修复但经历两轮——第一轮修了 4 个后还有一个失败，第二轮才修完。pushHistory 本质上是对"撤销后新操作应截断后续历史"的行为逻辑理解偏差。应在第一轮就分析所有 5 个失败的根本原因，统一设计方案而非分批处理。
