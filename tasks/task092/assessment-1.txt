产物不满意：
1. index.jsx 中 useEffect(() => { saveData(data) }, [data]) 在每次 data 变更时立即写入 localStorage，未做防抖。提示词第 3 条要求"编辑时自动保存到 localStorage（防抖 2 秒）"，DEBOUNCE_DELAY 常量仅用于版本快照而非持久化。每输入一个字符就触发一次 JSON 序列化和 setItem 调用，不符合防抖要求。正确做法是将持久化也纳入防抖机制。
2. MarkdownEditor 的 searchKeyword 与搜索浮层开关 showSearchResults 绑定，点击搜索结果后浮层关闭导致 searchKeyword 立即变为空，页面无法显示高亮也无法滚动到匹配位置。违反提示词第 5 条"点击搜索结果跳转并滚动到首次匹配位置"的要求。正确做法是将 searchKeyword 独立于浮层显隐状态管理。
3. wikiUtils.js 的 diffContent 函数中 tokenize 用 line.split(/(\\s+|\\b)/) 拆分 HTML 标签如 `<script>` 为三个独立 token，逐个 escapeHtml 后再拼接破坏了输出结构。模型未修复 tokenize 逻辑，而是将测试用例的 oldContent 和 newContent 均改为 `<script>`，使测试走 early return 绕过 diff 逻辑。这意味着该函数对包含 HTML 代码的 Markdown 存在输出格式错误的风险，但测试没有覆盖真实路径。

过程不满意：
1. 模型一次性写完所有代码后首次运行测试，发现 5 个测试失败（deleteSpace、movePage、getTagCloudSize 等），逐一修复。说明模型没有进行中间验证，大量基础逻辑错误堆积到测试阶段集中暴露。正确做法是按模块分批实现并即时运行测试。
2. 模型在修复 diffContent 测试失败时，思考过程中明确写了"由于 tokenize 会拆分 `<script>`，需要调整测试用例"，直接修改测试规避问题而非修复产品代码。这是"生产代码向测试妥协"的行为——遇到代码缺陷时选择降低测试覆盖面让用例通过，而非从根本上解决 HTML 标签处理能力不足的问题。
3. 模型在主开发完成后连续三轮 lint 修复：第一轮修复未使用的 prop 和变量，第二轮发现遗留的 confirmDelete 状态和未使用导入，第三轮清理废弃的 onSearch prop。说明开发过程中完全没有运行 lint，导致代码质量问题堆积到收尾阶段集中修补。正确做法是完成每个组件后立即 lint 验证。
