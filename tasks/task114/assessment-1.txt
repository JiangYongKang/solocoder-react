产物不满意：
1. buildGraphNodes 里依赖图边的方向搞反了。walk 函数遍历子依赖时用 `edges.push({ source: parentName, target: key })` 让父指向子。但需求说的是"被依赖方指向依赖方"，即子依赖指向父依赖才对。现在方向反了，用户看到的是谁依赖谁的流向完全颠倒。应该把 source 设成子依赖节点 key，target 设成父依赖 parentName。

2. Lock Diff 视图只覆盖了生产依赖的版本对比，devDependencies 完全漏了。index.jsx 传的 upgradedDependencies 是通过 applyUpgrades(dependencies, ...) 只对生产依赖算的，开发依赖压根没参与。用户切到 devDependencies 标签页选中包标记升级后，再切到 Lock Diff，这些开发依赖的变更在左右对比面板里完全不显示，变更统计也是零。应该把 dependencies 和 devDependencies 合并后一起传入构建。

3. DependencyTree 里的统计数字口径不一致。directDepsCount 取的是过滤前的直接依赖数，totalDepsCount 取的却是搜索过滤后的扁平结果数。用户搜关键词过滤后，顶部显示"共 N 个包（直接依赖 M 个）"，总数是过滤后的但直接依赖数还是过滤前的，两个数来自不同基准互相矛盾。应该统一统计来源，要么都过滤前要么都过滤后。

过程不满意：
1. 模型写完 DependencyTree.jsx 跑 ESLint 报解析错误，修了一处还报错，然后读文件看了一眼说"可能有隐藏字符"，用 Write 重写了整个文件才通过。就因为一处 JSX 模板字符串没闭合的语法错，经历了"初写→修补→重写"三轮。应该在写完文件后立即回读确认语法完整性。

2. 模型在 App.jsx 注册路由时两次编辑没改对：加 import、加 Route 后读文件发现"结构有问题"，又执行了第三次编辑修复。说明前两次编辑不够精确，可能插坏了已有 JSX 结构。对模板化程度高的路由配置，应该先通读完整上下文，import 和 Route 在一步精确操作中搞定。

3. 模型总结里写"可以通过 npm run dev 启动访问 /package-manager 查看效果"，这句话暗示实际根本没启动 dev server 验证。这个页面涉及树形表格递归展开、SVG 依赖图渲染交互、分栏同步滚动等大量 UI 逻辑，仅靠 65 个纯 JS 单测通过就说功能完成是不够的。应该至少启动一次 dev server 确认页面正常渲染、核心交互能走通。
