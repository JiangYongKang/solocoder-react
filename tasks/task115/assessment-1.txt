产物不满意：NoteCard 组件里声明了 isEditing 状态，handleDoubleClick 会把它设成 true，但整个组件的 JSX 渲染里没有一处读这个变量来切换编辑模式和查看模式。标题和内容的 textarea 始终无条件渲染且始终可编辑，用户不管单击还是双击看到的界面一模一样。提示词要求"便签支持双击直接编辑内容"，隐含意思应该是默认只读展示、双击切到可编辑。现在 isEditing 成了死代码，双击除了设一个没用状态外不产生任何视觉变化。应该默认用 span 或 p 展示内容，双击时设 isEditing 为 true 渲染 textarea，编辑完在失焦时切回只读。

过程不满意：模型规划时第一步就创建了七个 Todo，按"工具函数、测试、样式、主页面"的顺序实现。写完 1074 行的 index.jsx（内含六个内联子组件）后直接跑测试和构建，验证通过后就声称"双击编辑"已完成。整个开发过程没对 NoteCard 做过功能验收——比如在 isEditing 声明处搜索确认这个变量是不是真的被渲染逻辑消费了，也没启动 dev server 实际验证双击行为。说明模型自检阶段缺少逐条对照需求验证的习惯，只靠测试通过判断功能完成，导致死代码问题被漏掉。

产物不满意：getDropIndex 函数在计算拖拽插入位置时，没考虑被拖拽元素本身还占着 DOM 位置且带着 data-index。从左往右拖时被拖拽元素仍在 DOM 中被计入子元素数量，返回的索引比"移除被拖拽元素后"正确位置多 1。比如把 [A,B,C,D] 中 A 拖到 B 和 C 之间，getDropIndex 返回 2，moveNoteById 执行后变成 [B,C,A,D] 而不是期望的 [B,A,C,D]。从右往左拖则没这问题。应该在 fromIndex 小于 targetIndex 时把返回值减 1。

过程不满意：模型为 reorderNotes 和 moveNoteById 写了测试，但只测了"第一移到最后""中间移到开头"两种场景，没写从较早索引移到较晚索引的用例覆盖拖拽方向差异。而且测试只验证了工具函数本身的逻辑，没把 getDropIndex 的 DOM 计算和 moveNoteById 的数组操作联调。模型看到 88 个测试通过就判断拖拽搞定了，但函数测试通过不等于 UI 交互端到端正确。应该启动 dev server 实际拖拽便签验证视觉插入位置跟数组最终顺序是否一致。

产物不满意：handleSaveReminder 在用户点"清除提醒"后调 onSave(null)，把 reminderStatus 设成了 null 而不是常量里的 REMINDER_STATUS_PENDING。虽然 shouldTriggerReminder 通过先检查 reminderAt 是否为 null 防住了这个问题，但 reminderStatus 作为三态枚举里混进一个 null 破坏了语义完整性。同时 constants.js 定义了 REMINDER_STATUS_DISMISSED 却没在任何地方被导入使用，成了孤立常量。如果以后有代码只依赖这个字段判断业务逻辑，null 值会导致筛选遗漏。清除提醒时应该把状态重置回 PENDING，DISMISSED 状态也需要对应的设置入口。

过程不满意：模型导出三个提醒状态常量时一次写完了，但后续只实现了 PENDING 到 TRIGGERED 的转换，完全没有 DISMISSED 的设置入口——既没 markReminderDismissed 工具函数，NotificationToast 关闭按钮里也没调状态转换。写完 index.jsx 就测试构建声称完成，没回头检查 constants.js 里导出的每个常量是不是在业务代码里都有对应的读写路径。说明缺少"导出即使用"的自检步骤。应该完成后对所有导出常量做一次全局 grep 确认都被引用过。
