产物不满意：
模型修 copyToClipboard 测试时改了 routeUtils.js 的异常处理结构，把 navigator.clipboard 可用性判断从 try/catch 外面移进了里面。原因是测试里用了抛出异常的 getter 模拟 clipboard 不可用状态，异常发生在 try 块外部导致报错。虽然改动本身使 getter 异常可被捕获算合理增强，但触发它的根因是 mock 策略和产品代码原有结构不兼容，模型选了改产品代码而不是调整 mock 方案。根据规范遇到 mock 困难应调测试方案而非改生产代码架构，应该换用不抛异常的 mock 对象来模拟 clipboard 不可用。

过程不满意：
1. 模型完成 route-planner 测试后花大量时间调查 sticky-notes 包的测试状态，包括读源码、搜索函数定义、单独跑测试。sticky-notes 是另一个完全独立的任务包，跟 route-planner 毫无关系。规范规定只关注当前任务包代码，这种跨包探索就是范围越界，浪费了约两轮对话时间。

2. 模型写 downloadJSON 和 copyToClipboard 单测时经历了至少三轮测试文件全量重写：先追加内容发现缺 vi 导入，修了还失败又换局部编辑，报错后再 Write 全量重写，vi.stubGlobal 兼容性问题仍然失败，最后第二次 Write 全量重写才通过。整个过程是典型的试错式调试——没在动手前系统分析 jsdom 环境下这些浏览器 API 的正确 mock 方式，而是反复推翻方案重写整个文件。

3. 模型初期连续三次 view_files 读 routeUtils.test.js，中间没做任何修改操作，显示相同内容。之后分析 downloadJSON 又连续两次读 routeUtils.js。无意义的重复读取说明读了文件后没立刻行动，反复回去确认。正确做法是一次性读取后马上进入分析或修改阶段。
