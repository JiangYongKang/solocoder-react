产物不满意：
1. 修订痕迹标记只存在于侧边栏数据列表里，没渲染到文档编辑区的文本上。renderParagraphWithCursors 虽然调了 getRevisionsByParagraph 拿到了 revisions 变量，但后面根本没用它来标记段落文字。CSS 里定义了 .cd-revision-add/.cd-revision-delete/.cd-revision-format 样式，JSX 却从没把这些类应用到文本 span 上，也没渲染 tooltip。需求要求的绿色背景加下划线、红色背景加删除线、蓝色虚线边框全没做。应该在渲染时按 revisions 拆分文本为多个 span 并应用对应样式。

2. 格式修改修订类型从来没触发过。FORMAT 类型虽已在常量里定义且 CSS 有对应样式，但整个页面没任何代码路径能生成 FORMAT 修订。handleParagraphChange 只通过新旧内容长度对比判断 ADD/DELETE，工具栏没有格式化按钮，无法触发 FORMAT 修订。需求明确要求格式修改以蓝色虚线边框标记，这功能完全缺失。应该在工具栏加格式化按钮，修订模式下生成 FORMAT 记录。

3. 批注的文本选中功能没按需求实现。需求要求"选中一段文字后添加批注"，实际却用下拉框让用户选段落编号，批注原文取段落前 30 个字符。用户没法在编辑区划选具体文字来创建批注，产生的批注跟实际选中内容没关系。应该监听 selectionchange 事件获取用户划选范围，用选中文字填批注 text。

4. 光标位置不准确。moveRandomCursor 设了字符索引作为 offset，渲染时用 `left: ${Math.min(cursor.offset * 0.6, 95)}%` 把字符索引和容器宽度百分比强行线性映射，offset 为 10 和 100 对应的像素位置跟文字实际位置没关系。而且光标 span 插在 contentEditable 的 div 里跟文本混排，会干扰用户编辑。应该用独立叠加层通过 Range API 获取字符真实坐标来定位。

5. getCollaboratorById 里先用 findIndex 拿索引，再用 find 查同一用户，同一个数组遍历两次只为分别取索引和值。userId 等于 CURRENT_USER.id 时直接返回，idx 变量声明了但没用。应该合并为单次循环同时取索引和值。

6. 删除修订的比较逻辑太简陋了。handleParagraphChange 判断删除时只用 `newContent.length < oldPara.content.length` 并把多出长度全部标记为删除文字，这假定了修改只发生在末尾。用户在段落中间删字时，slice 截取的是末尾部分而不是实际被删的中间文字。应该用 diff 算法精确对比新旧内容找出真正被删的文字和位置。

过程不满意：
1. 模型开始后立即创建 7 项 Todo，没先通读 7 条需求做评估就规划。然后把所有功能塞进一个 740 行的 index.jsx，没有任何子组件拆分。项目里其他任务页面都做了组件拆分，模型没参考现有规范。单体组件导致功能高度耦合，修一个缺陷可能要通读整个文件。

2. 模型只跑了 npx vitest run 确认 82 个测试通过，从没启动 dev server 做页面实际渲染验证。因此没能发现修订痕迹没渲染到编辑区、光标偏差、批注无法选文字这些交互层缺陷。最终回复里说"全部实现完成"时压根没在浏览器验证过任何功能。应该启动 dev server 手工走通全部路径，逐条对照需求检查交互效果。

3. 光标定时器用了 setInterval(callback, 3000 + Math.random() * 5000)，Math.random() 在 useEffect 首次执行时就固化成一个值了，之后每次触发间隔都是这个固定值，不符合需求"每隔 3-8 秒随机移动"的要求。应该用 setTimeout 递归调用，每次执行完后重新算随机延迟。

4. 模型执行路径是"看项目结构→看示例页面→创建文件→跑测试→检查 lint"，全程没建立需求到实现的映射对照表，也没在完成后逐条回溯自检。导致修订渲染、划选批注、FORMAT 触发等多个功能点遗漏没及时发现。应该在实现前把需求拆成功能点清单，完成后逐项打勾确认。
