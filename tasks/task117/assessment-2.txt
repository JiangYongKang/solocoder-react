产物不满意：
1. handleFormat 在修订模式下为同一个格式化操作同时生成了 FORMAT 类型修订和多条 ADD 类型修订。用户选中文字点加粗时，handleFormat 先把选中文字包成 markdown 标记（如"**测试**"），再通过 processContentChangeWithRevision 检测内容变化。因为新旧内容确实不一样了，detectContentChanges 把新增的 `**` 标记符识别为新增内容，额外生成了 ADD 修订。结果一个格式化在修订面板里同时显示一条 FORMAT 加两条 ADD 修订，记录冗余语义混乱。应该在修订模式下不再追踪格式化实现细节的内容变化，标记符属于内部实现不应暴露为修订记录。

2. 非修订模式下格式化把 markdown 标记符直接注入段落内容造成视觉污染。handleFormat 无条件把选中文字包裹为 markdown 标记，但 renderParagraph 在无修订时直接渲染原始文本，不经过 renderContentSegments 即不应用格式化样式。用户点加粗后段落从"文字"变成"**文字**"，页面上显示的是带星号的原始文本。应该对格式化按钮增加 revisionMode 判断，非修订模式下禁用并提示，或把格式信息与内容分离存储。

过程不满意：
1. 模型收到提示词就创建了 7 个条目的 TodoList，然后才开始读代码。这时还没看 utils.js 和 index.jsx 的现有实现，不了解函数接口和组件结构，TodoList 基于对提示词的表面理解而非实际缺陷分析。应该先完整读现有代码理解现状和各缺陷根因，再基于代码现状定修复计划。

2. 模型读完两个核心文件后直接决定"完全重写 index.jsx"，用 Write 一次性替换了整个 950 行的组件文件。本轮是 Session 2 缺陷修复会话，提示词列出 6 个明确 BUG 修复点，每个有独立的问题描述和修复方向。完全重写让标注人员无法通过 git diff 清晰看每个 BUG 的具体修复内容，也存在引入新缺陷的风险。应该针对每个 BUG 做最小化 StrReplace 修改，每次修改有明确 diff 范围。
