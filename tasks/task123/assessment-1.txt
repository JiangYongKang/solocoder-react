产物不满意：
1. App.jsx 中缺少 FlashcardSystemPage 的 import、Route 路由和 TASKS 任务条目，从首页完全无法进入该二级页面。提示词要求代码放在 flashcard 目录已完成，但忽略了页面需集成到框架中可达的隐含前提。正确做法是完成功能代码后检查 App.jsx，确保 import、Route 和 TASKS 三处注册完整。

2. 标签数据格式在 CardEditModal.jsx 和工具函数间不一致，导致标签筛选完全失效。CardEditModal 将标签存为 {text, color} 对象格式，但 getUniqueTags 用 Set 存储对象引用，同内容不同对象被当作不同标签，filterCardsByTags 用 includes 做字符串比较永远不匹配。卡片列表渲染标签显示为 [object Object]，颜色也未用用户选择的值。正确做法是统一标签为纯字符串格式，颜色映射另存于常量中。

3. StudyMode.jsx 的 handleAnswer 在答完最后一张卡片时 recordStudySession 被重复调用。onFinish 捕获闭包中的旧 results（缺少最后一条记录），handleExitStudy 用不完整数据更新统计。finished 变为 true 后用户返回再调 onFinish 又更新一次，最后一张卡片被重复计入统计。正确做法是将新结果拼入后再调 onFinish，或用 useEffect 监听 finished 触发。

4. StatsPanel.jsx 热力图调用 getHeatmapCellColor 始终传入 count=0，所有日期显示同一浅绿色。getHeatmapCellColor 已实现四级颜色分级，但 StatsPanel 未利用 studyStats 计算每日学习量传入，分级功能形同虚设。正确做法是从 stats 提取每日 total 作为 count 传入，使热力图反映学习量差异。

过程不满意：
1. 模型完成代码和测试后未验证页面是否可达。直接声称功能覆盖，未检查 App.jsx 是否需要新增卡片，也未运行 pnpm run dev 验证。反映对任务边界理解偏差，只关注包内实现而忽略了页面需嵌入框架才能交付。正确做法是同步检查 App.jsx 的 import、Route 和 TASKS，并运行 dev server 做冒烟测试。

2. 模型修复 CardEditModal.jsx 的 lint 错误时经历试错式调试。第一次 Write 重写后发现 useRef 方案仍有问题，第二次又全量重写改为 key prop 控制挂载，未做最小修正。实际上只需将 useEffect 中 setState 改为 useState 初始化函数处理。这种反复推倒重来浪费效率。正确做法是先分析 lint 根因再针对性做最小改动。

3. 模型声称 85 个测试全部通过，但从未对标签存储为对象格式做端到端验证。测试中 filterCardsByTags 和 getUniqueTags 用例都用纯字符串数组，未覆盖对象格式标签实际场景。如果启动 dev server 手动创建带标签卡片，就会立即发现 [object Object] 问题。测试仅覆盖纯函数单元测试，缺少组件间数据流转的集成验证。
