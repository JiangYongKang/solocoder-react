产物不满意：
1. 模型没有在 App.jsx 中注册记忆卡片学习系统的页面路由和首页任务卡片。App.jsx 里缺少 `import FlashcardSystemPage from '@/pages/flashcard/index.jsx'` 导入语句，Routes 中没有 `<Route path="/flashcard" element={<FlashcardSystemPage />} />` 路由配置，TASKS 数组中也缺少对应的任务条目。导致从首页完全无法进入该二级页面，npm run dev 启动后点击不到任何入口，提示词第 6 条要求"代码实现需要放在 src/pages/flashcard/ 下"已完成但忽略了任务需集成到现有框架中可达这一隐含前提。正确做法是在完成功能代码后检查 App.jsx，确保新增的页面有对应的 import、Route 和 TASKS 任务条目，使页面能从首页卡片正常进入。
2. 标签数据格式在 CardEditModal.jsx 和工具函数之间不一致，导致标签筛选功能完全失效。CardEditModal 中 handleAddTag 将标签存储为 `{ text: '易错', color: '#ef4444' }` 对象格式，handleSaveCard 将此对象数组传入 createCard 存入 card.tags 字段。但 flashcardUtils.js 中的 getUniqueTags 函数对每个 tag 做 tagSet.add 后，Set 中存储的是对象引用，同内容的不同对象被当作不同标签。filterCardsByTags 使用 `card.tags.includes(tag)` 做字符串比较，对象永远匹配不上。CardList.jsx 渲染筛选 chip 时 tag 对象被直接渲染为 `[object Object]`，卡片列表中的标签也显示为 `[object Object]` 且颜色被硬编码为 `#6b7280` 而非使用用户选择的 tag.color。正确做法是统一标签为纯字符串格式，将颜色映射另存于常量或上下文中，使筛选逻辑能正确工作。
3. StudyMode.jsx 的 handleAnswer 函数在学习最后一张卡片时存在 recordStudySession 被重复调用的问题。当用户答完最后一张卡片后，handleAnswer 中 setResults 更新状态是异步的，但紧接着调用的 onFinish(results) 捕获的是闭包中的旧 results 值（缺少刚添加的最后一条记录），此时 handleExitStudy 用不完整的数据调用 recordStudySession 更新了一次统计。之后 finished 变为 true 页面重新渲染显示完成页面，用户点击"返回"按钮时再次调用 onFinish(results)（此时 results 已是完整的最新值），又触发 recordStudySession 再更新一次，导致最后一张卡片被重复累积到今日统计中。正确做法是在 handleAnswer 中判断为最后一张卡片时，将新结果直接拼入后再调用 onFinish，或使用 useEffect 监听 finished 状态变化来触发 onFinish。
4. StatsPanel.jsx 中热力图调用 getHeatmapCellColor 时始终传入默认 count=0，导致所有学习过的日期都显示同一种浅绿色 `#9be9a8`，无法按每日学习量分级展示不同深浅的绿色。getHeatmapCellColor 本身已经实现了 count 参数（0/1/2/更多对应四种不同绿色），但 StatsPanel 从未利用 studyStats 中的每日数据计算出 count 值传入，30 天热力图的四级颜色分级功能形同虚设。正确做法是从 stats 中提取每日的 total 值作为 count 参数传递给 getHeatmapCellColor，使热力图能反映每日学习量的差异。

过程不满意：
1. 模型在完成所有功能代码和测试后，没有执行最基本的"页面是否可达"验证。从 COT 记录看，模型在修复 lint 错误和运行测试后直接声称"功能覆盖"并给出全部勾选标记，既没有打开 App.jsx 检查任务卡片是否需要新增，也没有运行 pnpm run dev 在浏览器中验证页面是否能从首页进入。这反映出模型对"页面开发"任务边界的理解存在偏差——只关注了 src/pages/flashcard/ 包内的代码实现，忽略了二级页面需要嵌入到整体应用框架中才能交付这一前置条件。正确的做法是在完成包内代码后，同步检查 App.jsx 的 import、Route 和 TASKS 三处注册，必要时运行 dev server 做冒烟测试。
2. 模型修复 CardEditModal.jsx 的 lint 错误时经历了明显的试错式调试过程。COT 显示模型第一次用 Write 重写了该文件后发现 useRef 方案仍有问题，第二次又用 Write 全量重写改为通过父组件 key prop 控制挂载的方案，期间没有先缩小问题范围做最小修正。实际上只需将 useEffect 中 setState 的逻辑改为通过 useState 的初始化函数处理即可，不需要两次全量重写整个文件。这种反复推倒重来的行为模式浪费了开发效率，正确做法是先分析 lint 报错的具体原因（useEffect 内 setState），再针对性地做最小改动。
3. 模型在实验运行测试时发现中文字符串排序导致测试失败，修复了该测试用例后声称"所有 85 个测试全部通过"。但模型在整个过程中从未对标签存储为对象格式这一设计选择做过端到端测试验证——测试文件中 filterCardsByTags 和 getUniqueTags 的测试用例都使用纯字符串数组作为 tags，没有覆盖对象格式标签的实际运行场景。如果模型在写完 CardEditModal 后立即启动 dev server 手动创建一张带标签的卡片并查看卡片列表，就会立即发现标签显示为 `[object Object]` 的问题。这说明模型的测试策略仅覆盖了纯函数的单元测试，缺少对组件间数据流转的集成验证。
