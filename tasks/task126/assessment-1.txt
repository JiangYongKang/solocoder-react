产物不满意：
1. HistoryDrawer.jsx 中 formatCurrency 从 ./utils 导入但实际定义在 constants.js，运行时抛出 TypeError 导致组件崩溃、历史记录抽屉无法渲染。原因是未启动 dev server 做浏览器端验证，只依赖静态检查未发现路径错误。正确做法是改为从 constants.js 导入。

2. index.jsx 保存账单的前置条件仅校验参与者和费用数量，不验证费用是否填写完整。用户添加描述为空、金额为 0 或未选支付人的费用后可保存，无效条目被写入 localStorage。原因是 canSave 缺少 validateExpense 检查。正确做法是保存前遍历 expenses 调用 validateExpense，有错误则阻止并提示。

3. ExpensePanel.jsx 的 toggleSharedWith 增减分摊人时不考虑当前模式，一律将比例按均等重新覆盖。用户在自定义比例模式下精心设定百分比后增减分摊人，所有比例被重置需重填。原因是缺少对 expense.splitMode 的判断分支。正确做法是仅在平分模式下自动均分，自定义模式下新加入者默认 0%。

4. SettlementPanel.jsx 转账结算展示为「张三 → 李四 ¥35.50」，提示词要求「张三 需支付 李四 ¥35.50」。箭头虽能表达方向但缺少明确语义。正确做法是将箭头替换为文字"需支付"。

5. ParticipantPanel.jsx 调用 getAvatarColor 时将当前参与人数作为种子参数，而非仅基于姓名做确定性哈希。同一人名在人数变化时可能获得不同颜色，历史账单恢复后颜色漂移。正确做法是只用 name 参数计算哈希确保同姓名始终同颜色。

过程不满意：
1. 模型完成开发、Lint 检查和测试后声称全部通过，但全程未执行 pnpm run dev 做浏览器端验证。导致 formatCurrency 导入路径错误的运行时崩溃未被发现——它无法被 ESLint 或纯 JS 测试覆盖。正确做法是声明完成前启动 dev server 验证核心页面正常渲染。

2. 模型在项目探索阶段连续查看了 finance 目录下多个文件，但 finance 与 bill-splitter 任务没有任何关联。bill-splitter 页面严格独立隔离，翻阅无关页面代码属于无效探索。正确做法是仅了解技术栈即可直接开始实现。

3. 模型连续 3 轮修复才让测试通过。第二轮将 round2 相关测试期望从 -3.15 改为 -3.14 并修改比例分摊数值让断言通过，而非分析 round2 对负数的舍入行为是否符合财务惯例。这属于让测试通过而修改测试本身的试错模式。正确做法是先分析根因再做定向修正。
