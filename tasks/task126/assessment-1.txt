产物不满意：
1. HistoryDrawer.jsx 中 `formatCurrency` 从 `./utils` 导入，但该函数实际定义在 `constants.js` 中，`utils.js` 并未导出 `formatCurrency`。当用户打开历史记录抽屉查看已保存账单时，JSX 中调用 `formatCurrency(bill.totalAmount)` 会因为该标识符为 `undefined` 而抛出 `TypeError: formatCurrency is not a function`，导致整个 React 组件树崩溃、历史记录抽屉无法渲染。根本原因是开发完成时未启动 dev server 做浏览器端验证，只依赖了 ESLint 和 vitest 的静态检查，但 ESLint 的 `import/named` 规则未被触发而未报告此路径错误。正确做法是将导入改为 `import { formatCurrency } from './constants'` 并在 `HistoryDrawer.jsx` 中同步调整 `formatDateTime` 的导入路径。
2. index.jsx 中保存账单的前置条件 `canSave = participants.length >= 2 && expenses.length > 0` 仅校验了参与者和费用数量，完全不验证费用条目是否填写完整。当用户添加了一笔描述为空、金额填写为 0、或未选择支付人的费用条目后点击"保存账单"，系统会接受该无效条目并作为完整账单写入 localStorage。这对于一个账单分摊计算器是数据完整性的硬伤，用户恢复该账单后发现费用数据缺失，无法重新计算分摊结果。根本原因是 `canSave` 条件中缺少对 `validateExpense` 结果的检查。正确做法是在 `openSaveModal` 或 `handleSave` 中遍历 expenses 调用 `validateExpense`，若存在校验错误则阻止保存并提示用户修正。
3. ExpensePanel.jsx 的 `toggleSharedWith` 函数在用户增减分摊人时，不考虑当前处于平分还是自定义比例模式，一律将全部分摊人的比例按均等份额重新计算覆盖。当用户在自定义比例模式下针对每个人精心设定了百分比后因业务变化需要增减一个分摊人，所有已输入的比例被无条件重置为均分值，用户需从头逐项重填所有比例。根本原因是 `toggleSharedWith` 缺少对 `expense.splitMode` 的判断分支。正确做法是仅在平分模式下自动重新均分比例，自定义比例模式下对新加入的分摊人默认设为 0%，由用户手动调整。
4. SettlementPanel.jsx 中转账结算的展示格式为「张三 → 李四 ¥35.50」，而提示词要求「张三 需支付 李四 ¥35.50」。箭头符号虽然能表达方向，但缺少"需支付"这一明确语义后，用户可能误解箭头的方向含义，不符合提示词中对"谁欠谁多少钱"列表形式的显示规范。根本原因是结算条目渲染时只使用了 `→` 连接双方名称而遗漏了"需支付"文字。正确做法是在 `SettlementPanel.jsx` 的 `settlement-item` 渲染中，将箭头替换为文字"需支付"。
5. ParticipantPanel.jsx 中调用 `getAvatarColor(trimmed, participants.length)` 时，将当前参与者人数作为颜色计算的种子参数，而非仅基于姓名本身做确定性哈希。当同一参与者在人数规模不同的时刻被添加时（如先添加 3 人再全删后重新添加），可能获得不同的头像颜色，导致历史账单恢复后参与者的颜色发生漂移，视觉上造成同一人每次颜色不一致的混乱。根本原因是 `getAvatarColor` 的 seed 使用了不稳定参数。正确做法是通过只用 `name` 参数计算哈希而不传入 seed，确保同一姓名始终获得相同颜色。

过程不满意：
