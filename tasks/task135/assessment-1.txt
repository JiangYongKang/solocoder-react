产物不满意：
用户调节压缩质量滑块、尺寸缩放或切换输出格式后，index.jsx 中右侧"压缩后"预览面板始终显示占位文字"压缩后预览"，不会随参数变化自动更新压缩后的预览图。提示词第2条明确要求"参数调节后右侧实时展示压缩后预览和预估文件大小"，但 compressedDataUrl 只在点击"开始压缩"按钮后由 handleCompressSingle 函数中调用 compressImage 才生成，handleParamChange 函数仅更新了 params 状态和计算预计尺寸，未触发实时压缩。导致用户每次调整参数都要手动点击压缩按钮才能看到参数对图片质量的视觉影响，交互效率低下且丧失了实时调参的直觉反馈。正确做法是在 handleParamChange 中加入防抖逻辑，参数稳定后自动调用 compressImage 并将返回的 compressedDataUrl 和 compressedSize 更新到 selectedImage 状态中。
并排对比模式（side-by-side）下的 compare-info-grid 区域只分别显示了原图和压缩后图片的格式、尺寸、文件大小，完全缺少文件名的显示行。提示词第3条明确要求对比视图"下方分别显示文件名、格式、尺寸（宽×高）和文件大小（KB/MB）"，文件名信息在并排对比视图中缺失。当用户处理多张图片后逐一进入对比视图查看时，缺少文件名会导致无法快速区分当前对比卡片对应的是哪张图片的压缩结果。根本原因是 index.jsx 第674-713行的 compare-info-grid 渲染中两个 compare-info div 都只包含了格式、尺寸、大小的 info-row，遗漏了文件名的 info-row。解决方案是在每个 compare-info 中添加包含 selectedImage.name 的 info-row。
compressorUtils.js 的 calculateSavingsPercent 函数使用 toFixed(2) 保留2位小数精度计算节省百分比，而 storage.js 的 formatHistoryItemDisplay 函数内联实现了另一套节省百分比计算逻辑，使用 toFixed(1) 保留1位小数精度。同一张图片的压缩操作在完成后的结果面板中显示的节省百分比（calculateSavingsPercent输出）与在历史列表中显示的节省百分比（formatHistoryItemDisplay输出）数值不一致，例如实际节省66.67%在历史列表显示为66.7%，造成了数据前后矛盾。根本原因是 formatHistoryItemDisplay 未复用已有的 calculateSavingsPercent 工具函数，而是自行内联编写了计算逻辑并选择了不同的小数精度。解决方案是让 formatHistoryItemDisplay 直接调用 calculateSavingsPercent 获取节省百分比，确保全局数据展示统一。
compressImage 函数在压缩完成后通过 URL.createObjectURL(blob) 生成 compressedDataUrl，但这会在浏览器内存中创建一个临时 URL，当用户多次调整参数并重新压缩时前一次生成的 blob URL 未被调用 URL.revokeObjectURL 释放。虽然 blob URL 在页面关闭时会自动清理，但在单页应用的长时间使用中不断累积未释放的临时 URL 会造成浏览器内存泄漏。正确做法是在每次生成新的 compressedDataUrl 之前检查 selectedImage 上是否有旧的 compressedDataUrl（如果是以 blob: 为前缀），如果有则先调用 URL.revokeObjectURL 释放。
storage.js 的 lruEvict 函数名暗示其实现的是最近最少使用淘汰算法（LRU），但函数体只有一行 history.slice(0, maxItems)，实际行为是保留数组前 maxItems 个元素。由于 addToHistory 将最新记录插入数组头部，slice(0, maxItems) 的实际效果是保留最新 N 条记录并丢弃旧记录，这是简单的"保留最新"策略而非 LRU 淘汰。accessHistoryItem 函数虽然实现了将访问项移到数组头部的操作，但 lruEvict 和 addToHistory 均未调用 accessHistoryItem，导致整个历史记录的排序仅由添加顺序决定而非访问热度。正确做法是将 lruEvict 的实现与 accessHistoryItem 的访问序联动，按最近访问时间排序后淘汰最久未访问的项。

过程不满意：
模型在探索完项目结构后（第93行），尚未对提示词7条需求做逐条分析和功能要点拆解时，就调用 todo_write 创建了包含8个子任务的 TodoList 并立即开始编码。TodoList 条目如"实现图片压缩和批量处理"过于粗粒度，未细化到实时压缩预览、对比视图文件名展示等核心交互细节。这直接导致编码阶段遗漏了"参数调节后实时展示压缩后预览"这一需求，直到后期也未能在验证环节发现该缺失并补全。根本原因是模型在任务规划阶段对提示词的理解停留在表面关键词匹配，没有把每个需求句转化为可检验的子任务条目。正确做法是先通读提示词完整内容，逐条提取功能验收点并记录为可对照执行的 TodoList，然后再开始按清单逐项实现。
模型编写完全部代码和测试文件后第一次运行测试，终端输出15个测试用例失败。失败原因集中在两类：formatFileSize 函数对字符串类型参数未做 typeof 检查直接参与 Math.log 指数运算导致 `'0 B'` 预期不匹配实际返回值，以及测试文件未 mock localStorage 导致所有存储相关测试在 Node.js 环境中报 `localStorage is not defined` 错误。这两种问题在编写阶段都是完全可以预见的——工具函数对外暴露必然接收各种类型的输入，测试代码运行在 Node 环境天然没有浏览器 localStorage。根源在于模型编码时缺乏防御性编程意识，对每个函数没有在上线前做"入参如果是非预期类型会怎样"的预推演。正确做法是在编写工具函数时主动考虑空值、类型错误、边界值等异常场景，在测试文件编写时识别出需要 mock 的浏览器 API 并在文件顶部集中声明。
模型运行 eslint 检查 image-compressor 目录首次发现6个 lint 错误后，只针对性修复了其中部分问题（storage.js 中未使用的行参数 context），但未做全局排查。再次运行 eslint 后又出现新的错误，包括测试文件中 catch 块的 error 参数在部分地方未使用以及 compressorUtils.js 中 compareModes 常量已导出但未被任何文件导入。整个 lint 修复过程走了两到三轮才最终清除所有错误。根本原因是第一轮修复时没有通览所有 lint 输出并制定完整的修复计划，而是零散地读一个改一个。正确做法是在首次获取完整 lint 输出后先通读所有错误条目，列出完整的修复清单，一次性处理所有问题后再验证。
