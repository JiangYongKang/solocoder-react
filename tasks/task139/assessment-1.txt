产物不满意：
1. textDiffStats.js 的 computeSimilarity 使用 `(lcsLength / maxTotal) * 100` 计算相似度，而提示词第 3 条指定公式为 `1 - (差异数 / 两端较大总数) × 100%`。当两端长度不等时两个公式结果不同——例如 "abcd" 与 "axy" 对比，模型得 25%，提示词公式得 -25%。模型自行改变公式没做任何说明，导致非对称文本场景下数值不一致。应该严格按照提示词的指定公式实现。
2. 差异高亮颜色与提示词第 4 条不符。提示词要求行内差异和整行差异统一用红色背景标记，但代码中新增行用绿色、删除行用红色、修改行用黄色，模型自行引入了三色体系替代纯红色标记。应该是所有差异标记统一使用红色系。
3. index.jsx 声明了 rightDiffRef 但未绑定到任何 DOM 元素，handleScrollSync 中 sourceSide 为 'left-diff' 时会尝试设置 rightDiffRef，但该 ref 永远为 null，同步滚动逻辑完全失效。虽然因使用单一 grid 容器所以实际不需要同步，但保留这段永不执行的死代码属于未清理的遗留逻辑。应该删除 rightDiffRef 及相关死代码。
4. 缺少对 storage.js 的 localStorage 持久化函数的单元测试。提示词第 7 条要求测试放在 __tests__/text-diff/ 下，该目录只有 textDiffStats 和 diffUtils 的测试，没有 storage 测试。storage.js 的 loadHistory、saveHistory、addHistoryItem 等函数都是纯 JS 逻辑需测试其去重、最大条数限制等行为。应该补充 storage.js 的单元测试。
5. textDiffStats.js 与已有的 diffUtils.js 存在功能重复。splitIntoLines 与 splitLines 几乎相同，computeCharDiffForLine 与 computeCharDiff 算法一致，buildLineDiffPairs 返回的 type 只有 equal、added、removed，但 renderLineDiff 中判断了 modified 分支永远无法命中——因为没像 diffUtils 的 mergeModifiedLines 那样合并相邻增删行。应该优先复用 diffUtils.js 已有的算法函数而非重写。

过程不满意：
1. 模型收到任务后第一个动作就是建 TodoList，完全没有先分析提示词的 7 条需求，也没阅读已有代码和 diffUtils.js。建完 TodoList 后才查看项目结构和文件，导致规划时对项目上下文一无所知——不知道已有 diff 算法库、不了解代码架构，致使后续全量重写而非增量扩展。应该先通读需求和巡查已有代码再制定计划。
2. 模型在查看 diffUtils.js 后发现了已有完整的 diff 算法工具函数，却选择在 textDiffStats.js 中重新实现了一套功能重复的函数。这种另起炉灶的做法直接导致了死代码路径（永远无法进入的 modified 分支）和代码冗余。应该分析已有代码的接口，对不符合需求的部分做最小化修改和包装。
3. 模型在完成开发后跑 eslint 发现错误，随后进行了三轮反复修改，第三轮中仍遗留了 useEffect 导入未删除的问题需要第四轮修复。同一 index.jsx 文件在 lint 修复中被读写四次，说明首次提交的代码质量偏低，应该在编写时就注意避免常见 lint 问题而非依赖事后修复。
