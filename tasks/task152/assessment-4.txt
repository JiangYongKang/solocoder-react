产物不满意：模型新增的两个测试用例都用 Set 去重追踪文件节点 type 访问次数。Set 去重意味着即使某文件被重复访问十次也只保留一次，测试实际验证的是"每个文件至少被访问过一次"而非名称的"每个节点只遍历一次"。当 buildDirectoryCoverage 因代码变更导致子树被重复遍历时，测试检测不到。原因是在发现 type 被多次读取后，模型没分析是否合理，直接把策略改为 Set 去重让测试通过，属于测试向实现妥协的反模式。

过程不满意：模型收到 prompt-4 后第一步就创建了 TodoList，还没读代码就重复了 assessment-3 指出的先规划后阅读问题，没从标注反馈学习。其次 getter 方案测试首次失败后，发现 type 被多次读取却没诊断是否合理——代码中 current.type 两次判断属正常控制流——直接改为 Set 去重，语义从"只遍历一次"退化为"至少访问过一次"。没尝试 Map 累加计数，遇到失败就改测试策略而非先诊断根因，是试错式调试，还削弱了测试验证力度。
