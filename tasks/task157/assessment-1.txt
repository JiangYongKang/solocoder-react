产物不满意：
1. abTestCore.js 中导出了 `regularizedIncompleteBeta`、`betaCF`、`gammaLn` 三个完整的统计学函数（合计约100行代码），但 `calculatePValue` 并未调用它们，而是使用了一套手写的分段线性近似来计算 p 值。这三个函数在文件中完全不被引用，构成死代码，且它们内部还有嵌套的函数调用链（如 `regularizedIncompleteBeta` 调用 `betaCF` 和 `gammaLn`），增加了代码的理解和维护负担。原因是模型先实现了 Beta 分布精确算法但测试不通过，之后改用分段近似替代，却忘记清理旧的算法实现。正确做法是移除未使用的三个函数，或者改造 `calculatePValue` 调用 `regularizedIncompleteBeta` 来实现统计学上更准确的 p 值计算。
2. updateTimeSeriesData 函数（abTestCore.js）的执行逻辑是 `[...currentData.slice(0, -1), lastEntry]`，即每次定时器触发时只替换时间序列中最后一天的数据点，而非在序列末尾追加新的一天。这导致无论实验运行多久，折线图上始终只有创建时的 7 天数据，实时数据刷新的效果完全无法体现——用户看不到数据随时间增长的动态过程，第四个提示词需求「数据用定时器模拟周期性刷新」的实际效果大打折扣。原因是实现时把"更新最新数据点"和"追加新数据点"两个概念混淆了。正确做法是追加新条目而非替换旧条目。
3. constants.js 中定义了 `METRIC_KEYS` 并导出，但搜索整个 ab-test 包的所有文件，该常量从未被任何地方 import 或引用。这是一个无实际用途的冗余导出，且该常量仅是对 `METRICS` 数组做了一次 `.map((m) => m.key)` 的派生结果，即使未来需要也应在调用处实时计算而非预先导出全局常量。原因是编写 constants.js 时顺手定义了辅助变量，但后续开发中未找到使用场景也未清理。
4. index.jsx 中 ExperimentDetail 组件内的 `useEffect(() => { setCurrentExperiment(experiment) }, [experiment])` 是冗余的副作用。`currentExperiment` 的初始值已经通过 `useState(experiment)` 设为了 prop 的值，当父组件传入新的 `experiment` 时组件自然会重新挂载或用 key 触发重置，这个 effect 不仅多余还引发了 ESLint 的 `react-hooks/set-state-in-effect` 错误（当前 lint 检查 ABTestPage 和 ExperimentDetail 中各有 1 个同类错误未修复）。正确做法是删除该 effect，需要同步外部 prop 变化时通过给组件加 `key={experiment.id}` 来让 React 自然重建组件实例。

过程不满意：
1. 模型在首次运行测试后发现测试失败，随后对 abTestCore.js 发起了至少 7 轮反复修改，包括调整流量分配算法的 rounding 误差修正逻辑、修改 calculatePValue 的分段参数值、以及修复状态机返回逻辑。每次修改后模型都先读取整个文件查看效果再运行测试，形成"改-读-测-再改"的循环。尤其在修复 p 值计算的分段近似参数时，模型连续尝试了多组不同的数值组合而未先分析测试断言中"期望值 vs 实际值"的偏差模式。这种不分析根因而反复尝试的试错式调试浪费了大量推理资源，同一类错误出现两轮以上仍未修正根本原因是强信号。正确做法是在首次测试失败后暂停修改，先逐条分析失败断言的期望值和实际值差异，定位是边界条件、算法设计还是数值精度问题，然后一次性修正。
2. 模型在完成 lint 修复阶段后，总结中明确写道"核心逻辑文件（abTestCore.js、storage.js、test 文件）lint 检查通过"，但实际运行 lint 检查发现 index.jsx 仍有 2 个 `react-hooks/set-state-in-effect` 错误。问题出在模型进入 lint 修复时先查看了 index.jsx，但只关注并修复了 `controlGroup` 变量未使用的问题，在再次查看文件确认修复后就直接结束了——整个过程中始终没有运行过针对 index.jsx 的完整 lint 检查，也忽略了文件中已存在的 useEffect 内 setState 警告。正确做法是在所有文件修改完毕后，对 ab-test 包下全部源文件统一运行一次 ESLint，审阅完整输出而非只看第一处错误就停止。
3. 模型在开始实现前查看了 `src/pages/survey/index.jsx` 和 `src/pages/__tests__/chat/utils.test.js` 来"了解项目结构和编码规范"，但这两个文件与 ab-test 任务完全无关（分属 survey 和 chat 包）。这种跨包参考在标注规范中是违规行为——每个任务的所有代码应集中在语义对应的单一包下，审查和实现都不应涉及其他任务包。虽然只是查看操作而非代码复制，但说明模型对任务隔离原则的边界意识不足。正确做法是只参考项目根目录的 package.json 等全局配置文件来了解技术栈，编码规范的参考应限于 ab-test 包自身。
