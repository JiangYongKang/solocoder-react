产物不满意：
1. DanmakuArea.jsx 的第一个 useEffect 用 for 循环创建了 6 个 setTimeout 分批发送初始评论弹幕，cleanup 函数中却没有任何 clearTimeout 调用。当用户快速进入直播间后立即退出页面时，组件卸载后这些 setTimeout 的回调仍会执行 addDanmaku 进而触发 setState，React 开发模式下会在控制台输出 "Can't perform a React state update on an unmounted component" 警告。根本原因是 useEffect 只关注了如何启动副作用而忽略了对称的清理逻辑，且模型在重写 DanmakuArea.jsx 时仍然没有补充清理代码，两次编写都遗漏了同一个问题。应在 useEffect 中创建数组收集所有 timeout ID，在 cleanup 函数中遍历 clearTimeout 全部清除。
2. index.jsx 中给 ShoppingBag 添加了 key={currentProduct?.id} 使组件在讲解商品切换时被销毁后重建。副作用是 selectedSpecs 和 addQuantity 等组件内部状态在商品切换时被重置为初始值。轮播每 3 秒自动切换商品时，如果用户正在购物袋中浏览当前商品的规格但尚未点击加入，切换后所有临时选择被静默丢弃。根本原因是模型用 key 强制 remount 来规避 ESLint React Hooks exhaust-deps 警告，而非在 ShoppingBag 内部通过 useEffect 监听 currentProduct 变化来手动重置相关状态。这种方案牺牲了用户在商品切换过程中的交互连续性。
3. LiveWidgets.jsx 的 LikeButton 组件中 handleClick 计算爱心出现位置的逻辑存在坐标偏移。代码通过 e.currentTarget.offsetParent 获取 .ls-like-container 的坐标，将按钮中心减去容器左上角得到 x 和 y 值，然后将爱心以 left:x top:y 定位在 .ls-hearts-layer 内。但 .ls-hearts-layer 的 CSS 设置了 inset:-300px -100px 0 -100px，即该层的坐标原点比 .ls-like-container 左上角偏左 100px 偏上 300px。导致爱心实际出现位置与按钮中心相差约向左 100px 向上 300px，用户点击时爱心并非从按钮正中心飘出。根本原因是计算坐标时未考虑 hearts-layer 的 inset 偏移量。应在 CSS 中将 inset 统一为 0，让 .ls-hearts-layer 与 .ls-like-container 的坐标系保持一致，或者对计算的 x、y 值额外加上 inset 偏移补偿。
过程不满意：
1. 模型在完成所有组件和 CSS 的编写后才首次运行 ESLint，此时发现了大量 JSX style 对象中像素值未用引号包裹的基础语法错误，如 {width:12px} 应写为 {width:'12px'}。这些错误分布在多个文件中，导致模型不得不整体重写 DanmakuArea.jsx 和 ShoppingBag.jsx 两个组件。重写后 ShoppingBag 又出现了 React Hooks 依赖问题需要第三轮修改才解决。如果模型在每完成一个组件后就运行 lint 检查，这些基础语法错误会在第一时间被发现和修正，不会积累到需要两个文件全文替换的程度。
2. 模型在遇到 ESLint 报告的 React Hooks exhaust-deps 警告时，直接选择了用 key={currentProduct?.id} 让 ShoppingBag 强制 remount 的方案。思考过程中仅记录了 "用 key 属性让 ShoppingBag 在商品切换时自动重挂载是更好的方式"，没有分析此方案会导致组件内部状态在轮播自动切换时丢失的副作用，也没有讨论 useEffect 监听 currentProduct 变化的替代方案。这反映出模型在面对 lint 警告时倾向于选择最快消除警告的捷径，而非深入分析问题根因和评估不同方案对用户体验的影响。
3. DanmakuArea.jsx 的第一个 useEffect 使用了 setTimeout 循环但 cleanup 函数为空的问题，在模型的重写过程中自始至终没有被发现和修复。模型在首次编写和整文件重写 DanmakuArea.jsx 时都只关注了如何启动弹幕生成逻辑，却没有遵循 React 副作用清理的对称性原则为 setTimeout 添加对应的 clearTimeout。这说明模型对 useEffect 的 cleanup 机制理解不够深入，没有形成"每个需要清理的副作用都必须有对应清理代码"的编码习惯。
